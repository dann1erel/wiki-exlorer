# 2. Описание проекта

Проект **Wiki Explorer** представляет собой CLI-утилиту для получения и анализа информации из Wikipedia. Работа с программой выполняется через командную строку: пользователь вводит название команды, необходимые аргументы и дополнительные опции.

Основная идея проекта заключается в том, чтобы предоставить удобный инструмент для быстрого анализа статей Wikipedia без открытия браузера. Утилита может быть полезна для учебных целей, первичного анализа тем, просмотра связей между статьями и получения статистической информации.

В проекте предусмотрен следующий набор команд:

- `search` — поиск статей Wikipedia по ключевому запросу;
    
- `info` — получение основной информации о статье;
    
- `links` — вывод внутренних ссылок статьи;
    
- `graph` — построение графа связей между статьями;
    
- `categories` — получение категорий статьи;
    
- `pageviews` — получение статистики просмотров статьи;
    
- `random` — получение случайной статьи;
    
- `images` — получение и скачивание изображений из статьи.
    

Таким образом, проект соответствует требованию о наличии минимум семи команд или подкоманд.

Для работы с внешними данными используются два основных API.

**MediaWiki Action API** применяется для большинства команд проекта. Через него выполняется поиск статей, получение информации о страницах, внутренних ссылок, категорий, изображений и случайных статей.

**Wikimedia Pageviews API** используется для команды `pageviews`. С его помощью программа получает статистику просмотров статьи за выбранный период.

В репозитории представлены две реализации одной и той же идеи. Это позволило сравнить не только итоговый код, но и процесс разработки с разными ИИ-инструментами. Для каждой версии проекта была создана отдельная папка, собственная структура файлов и собственный набор тестов.

# 6. Разработка версии ChatGPT

## 6.1. Общий процесс разработки

Разработка версии `version_ChatGPT` велась поэтапно. На первом этапе с использованием ChatGPT было подготовлено техническое задание, в котором были описаны функциональные требования, команды CLI-утилиты, параметры команд, используемые REST API, структура проекта, обработка ошибок, тестирование, конфигурация и логирование.

После подготовки технического задания процесс разработки был разделён на отдельные этапы. Каждая команда CLI рассматривалась как самостоятельная задача: сначала формулировался промпт для реализации конкретной функциональности, затем генерировался код, после чего он добавлялся в проект и проверялся через запуск в терминале.

Важной особенностью работы с ChatGPT стало использование приёма **метапромптинга**. На отдельных этапах нейросеть использовалась не только для непосредственной генерации кода, но и для подготовки более подробных промптов на основе технического задания. Такой подход позволял уточнить требования перед реализацией конкретной команды и снизить вероятность того, что модель пропустит важные детали.

Общая схема работы выглядела следующим образом:

- подготовка технического задания;
    
- генерация подробного промпта для реализации конкретной команды;
    
- генерация кода по подготовленному промпту;
    
- добавление кода в проект;
    
- запуск команды в терминале;
    
- фиксация ошибок;
    
- передача текста ошибки в диалог с нейросетью;
    
- получение исправлений;
    
- повторная проверка результата.
    

Такой подход позволил сделать процесс разработки более управляемым. Вместо попытки сгенерировать весь проект целиком работа была разбита на небольшие этапы: реализация команд `search`, `info`, `links`, `graph`, `categories`, `pageviews`, `random`, `images`, добавление тестов, подготовка README и проверка соответствия техническому заданию.

## 6.3. Возникшие проблемы при работе с ChatGPT

Использование метапромптинга также уменьшило количество неудачных запросов. Когда промпт предварительно уточнялся на основе ТЗ, итоговая задача для генерации кода становилась более конкретной. Особенно полезным этот подход оказался при переходе от одной команды к другой.

## 9.3. Рекомендации по промпт-инжинирингу

В процессе работы стало понятно, что качество результата сильно зависит от качества промпта. Чем точнее формулируется задача, тем меньше ошибок возникает на этапе реализации.

Во-вторых, полезно прикладывать техническое задание или его фрагмент. Это помогает нейросети учитывать конкретные требования: названия команд, параметры, ожидаемое поведение, используемые библиотеки и формат вывода.

В-третьих, перед генерацией кода оказался полезен приём метапромптинга. Сначала нейросеть использовалась для составления более точного промпта на основе ТЗ, а затем этот промпт применялся для реализации конкретной части проекта. Такой подход помогал уточнить задачу и уменьшить количество пропущенных требований.

В-четвёртых, при возникновении ошибки важно передавать в диалог полный текст ошибки, а не только краткое описание. Traceback, команда запуска и текущая структура проекта помогают нейросети точнее определить причину проблемы.

В-пятых, после генерации кода необходимо обязательно выполнять ручную проверку: запускать команды, тесты, сверять параметры с ТЗ и проверять примеры использования. ИИ может ускорить разработку, но не заменяет полноценную проверку результата.
